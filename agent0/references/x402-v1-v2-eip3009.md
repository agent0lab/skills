# x402 v1/v2 and EIP-3009

## x402 versions

- **v1** — Older format; payment requirements in header or body; may use X-PAYMENT or similar.
- **v2** — Uses PAYMENT-REQUIRED header (base64 JSON) and PAYMENT-SIGNATURE for the payment payload. Supports ResourceInfo and CAIP-2 chain identifiers (e.g. `eip155:84532`).

The SDK parses both. **X402Accept** normalizes fields (e.g. `amount`/`payTo` → `price`/`destination`). **x402Version** on the 402 response indicates which version the server used.

## EIP-3009 (TransferWithAuthorization)

For EVM payment options, the SDK builds a signed EIP-3009 TransferWithAuthorization payload. The signer (private key or wallet) must hold the token and have approved the transfer. The server (or a verifying contract) validates the signature and completes the transfer.

Non-EVM accepts (e.g. Solana) are **filtered out** when the SDK builds the payment response, so only EVM options are surfaced to the caller. If the server returns only non-EVM options, pay() will have no EVM option to use.

## PAYMENT-SIGNATURE header

After building the payment, the SDK retries the original request with a **PAYMENT-SIGNATURE** header (or, for v1, the header name the server expects). The value is typically a base64-encoded payload containing the signed authorization and metadata.

## Summary

- Use **x402Version** from the 402 response when you need to distinguish server behavior.
- For EVM, the SDK uses EIP-3009; ensure signer has token and allowance.
- Non-EVM accepts are filtered out; only EVM payment paths are used by `pay()` / `payFirst()`.
